# METADATA KNOWLEDGE BASE

## OVERVIEW
`metadata/` holds overlay-wide config plus version-keyed cache data; it is structured repo metadata, not package implementation code.

## STRUCTURE
```text
metadata/
├── layout.conf    # overlay inheritance + manifest policy
└── md5-cache/     # version-keyed package cache entries
```

## WHERE TO LOOK
| Task | Location | Notes |
|------|----------|-------|
| Overlay inheritance / manifest policy | `metadata/layout.conf` | Source of `masters`, `thin-manifests`, and manifest-signing settings. |
| Package cache entry | `metadata/md5-cache/<category>/<pkg-version>` | Cache files are version-keyed filenames, not bare package dirs. |
| Cache shape examples | `metadata/md5-cache/app-emulation/wine-osu-*` | One package can have multiple versioned entries. |
| Cache shape examples | `metadata/md5-cache/sys-kernel/cachyos-sources-*` | Same version-keyed pattern in kernel packages. |

## CONVENTIONS
- `layout.conf` is the authoritative overlay-policy file in this subtree.
- `md5-cache/` is organized as structured, package-version-keyed entries such as `app-emulation/wine-osu-10.3-r1`.
- The cache subtree is machine-shaped and regular; treat it as structured data, not prose.
- Evidence here supports cautious edits, but not blanket claims that the cache is autogenerated.

## ANTI-PATTERNS
- Do not refer to non-versioned `md5-cache/<category>/<package>` paths when the tracked entries are version-specific.
- Do not duplicate overlay inheritance or manifest-policy settings outside `layout.conf` without a concrete reason.
- Do not describe `md5-cache/` as generated unless you have stronger proof than naming/layout alone.
- Do not use `metadata/` as a catch-all place for package-specific notes.

## NOTES
- If a task is really about an ebuild or payload, move back to the owning category/package tree; `metadata/` is the overlay-policy/cache layer.
- Changes under `metadata/**` do not exercise the changed-ebuild PR validation path by themselves; if you need CI confidence here, use a verification path other than waiting for package-dir PR checks.
